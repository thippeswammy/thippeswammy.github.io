## 📖 Overview

This repository builds upon foundational behavioral modeling to present a complete **Vehicle Controller** for a Skateboard EV platform, developed in our SDV Lab. 

The core challenge and driving philosophy behind this project was: **What if we assume there are absolutely no physical sensors or ECUs present on the vehicle?** Instead of relying on physical hardware constraints, this project leverages **MATLAB/Simulink** and the **dSPACE SCALEXIO** system to do the heavy lifting. By defining a model that handles every single signal the vehicle produces or needs, dSPACE completely emulates the physical world for the controller. It acts not just as a testing monitor, but as the virtual sensors and ECU environment, allowing for full external control and deep behavioral analysis.

## 🛠️ Project Architecture & Testing Phases

The project is structured around utilizing dSPACE as a complete digital twin for the missing physical environment:

### 1. Total Simulation Modeling (MATLAB/Simulink)
Developed a comprehensive plant model representing the Skateboard EV's dynamics. Unlike standard models, this was specifically engineered to map every single signal the vehicle would produce or need, allowing total external control of those functions.

### 2. Software-in-the-Loop (SIL) & Sensor Emulation
* **Virtual Environment Setup:** Instead of reading physical sensors, dSPACE was configured to generate and inject the exact real-time data the controller expected. 
* **Signal Looping:** By looping these signals internally, the system successfully tricked the controller into believing it was operating in the physical world, validating the logic safely in software.

### 3. Hardware-in-the-Loop (HIL) Deployment & Telemetry
The transition to real-time execution:
* **Bridging Simulation & Reality:** We injected external acceleration commands through dSPACE to drive the system via CAN.
* **Real-World Observation:** We successfully pulled live telemetry from the physical world, allowing us to see exactly how the physical components responded to the simulated control logic.

## ⚙️ Vehicle Controller Specifications

The core logic processes external commands and translates them into completely emulated sensor feedback and control parameters using standard automotive communication protocols.

| Feature | Input/Emulation | Output Behavior | Functional Description |
|---------|-----------------|-----------------|------------------------|
| **CAN Communication** | Bit-level signal manipulation | Formatted CAN Messages | Emulating the vehicle's internal network to send and receive accurate, bit-precise control signals. |
| **Virtual Sensor Injection** | dSPACE Generated Signals | Emulated Data Streams | Emulating the presence of physical sensors to feed the controller the exact network data it expects. |
| **External Command Injection** | User Input via ControlDesk | Target Control Vectors | External commands (like acceleration) injected directly into the Simulink model to bypass physical throttles. |
| **Real-Time Telemetry** | Physical world feedback | ControlDesk Dashboard | Capturing and visualizing live data from the physical system to monitor how the controller is interacting with the real world. |

### 🚀 Extensibility (What's Next?)
Because the system is already successfully looping real-world **Telemetry**, the foundation is set for advanced physical analysis. Future extensions of this project can easily include:
* **Motor Startup Logic Analysis:** Comparing the theoretical start curve generated by the Simulink model against the actual physical startup.
* **Physical Motor Profiling:** Logging real-world current draw, torque response, and acceleration times to fine-tune the controller's algorithms.

## 🔌 dSPACE SCALEXIO Signal Mapping

Because dSPACE is emulating the physical world, the pinout focuses on the generated signals mapped via ConfigurationDesk.

**Virtual Inputs (dSPACE to Controller):**
* **Emulated Sensor Feeds:** Speed, battery state, and assumed physical conditions.
* **External Control Triggers:** Acceleration and braking commands injected via ControlDesk.

**Physical Outputs (Controller to Skateboard EV):**
* **CAN Bus Lines:** Sending bit-level control messages to actuate the physical skateboard motor.
* **Telemetry Loopback:** Feeding real physical responses back into dSPACE for live monitoring and logging.
